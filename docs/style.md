# C언어 코딩 규격 (Style Guide)

프로젝트의 안정성과 유지보수성을 위해 다음의 코딩 규격을 엄격히 준수합니다.

## 1. 명명 규칙 (Naming Conventions)
* **변수 및 함수:** `snake_case` 사용 (예: `check_collision`)
* **구조체 및 타입:** `_t` 접미사 사용 (예: `entity_t`)
* **포인터 변수:** `p_` 접두어 사용 (예: `entity_t *p_target`)
* **전역 변수:** 상태 관리를 위해 불가피한 경우에만 `g_` 접두어 사용 (예: `g_game_state`)
* **상수(매크로/Enum):** `UPPER_SNAKE_CASE` 사용

## 2. 함수 설계 및 에러 처리 (Error Handling)
* 모든 주요 로직 함수는 에러 코드(`int`)를 반환하여 호출자에게 성공/실패 여부를 알립니다 (성공: 0, 실패: 음수).
* 실제 반환해야 할 데이터는 함수의 인자(포인터)를 통해 아웃풋 매개변수로 전달합니다.
* 메모리 누수를 막기 위해, 함수 내 예외 상황 발생 시 `goto ERROR_EXIT;` 레이블로 이동하여 할당된 자원을 일괄 해제하는 방식을 권장합니다.

## 3. 방어적 코딩 (Defensive Programming)
* 함수의 시작 부분에서 포인터 매개변수가 `NULL`인지 반드시 검증합니다.
* 값이 변경되지 않아야 하는 매개변수나 포인터에는 `const` 키워드를 적극적으로 사용합니다.
* `malloc` 등 동적 메모리 할당 직후에는 반드시 할당 성공 여부를 검사하며, 더 이상 쓰이지 않을 때 명시적으로 `free`를 호출합니다.
* **구조체 초기화 함수(`_init`)는 멤버 전체를 명시적으로 초기화해야 합니다.** 스택에 할당된 구조체는 쓰레기값을 포함할 수 있으며, 멤버 초기화를 생략하면 구조체에 새 필드가 추가될 때 해당 필드가 초기화되지 않은 채 남아 잠재적 버그가 됩니다. `memset` 또는 명시적 대입으로 모든 멤버를 초기화하십시오.
  * **실제 사례:** `monster_t`에 필드를 추가하면서 `game_state_t` 레이아웃이 바뀌었고, `turn_manager_init()`이 `shop.active`를 초기화하지 않아 스택 쓰레기값이 남아 상점 UI가 항상 활성화되는 버그가 발생했습니다.
